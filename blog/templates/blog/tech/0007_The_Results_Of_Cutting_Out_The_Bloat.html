The Results of Cutting Out The Bloat

A while back, I wrote about cutting out the bloat in web development. The idea was simple: stop using massive frameworks for everything and get back to basics. Now that this blog has been running for a while, I wanted to share the actual results of that philosophy. Spoiler alert: they're pretty damn good.

The Numbers Don't Lie

Let's start with the hard data. This site consistently scores in the high 90s on Lighthouse audits across all four metrics:

  • Performance: 95-100
  • Accessibility: 95-100
  • Best Practices: 95-100
  • SEO: 95-100

For context, the average website scores around 50-70 on performance. Many popular blogging platforms struggle to break 80. Meanwhile, this "simple" Django site with minimal JavaScript and no frontend framework regularly hits perfect or near-perfect scores.

But scores are just numbers. What do they actually mean in the real world?

What Users Actually Experience

The performance metrics translate to real benefits:

  • First Contentful Paint: Under 0.5 seconds - Users see content almost instantly
  • Time to Interactive: Under 1 second - The page is fully usable immediately
  • Total Page Size: Under 200KB for most pages - Uses less data than a single high-res image on most modern sites
  • Zero JavaScript frameworks: No React, no Vue, no Next.js bloat to download and parse

Compare this to the typical blog built with modern frameworks: 2-3MB of JavaScript, 3-5 second load times, and a spinning loader while the framework bootstraps. Is that really necessary to display text and images?

The Developer Experience

Here's where it gets really interesting. Building without the bloat isn't just good for users - it's fantastic for developers too.

Build Times

There is no build step for most of the site. I write HTML, save the file, refresh the page. That's it. No waiting for webpack to compile, no hot module replacement failures, no build cache invalidation issues.

When I do need to optimize CSS or JavaScript, it's a simple one-command process that takes seconds, not minutes.

Deployment

The entire deployment process is straightforward: push to git, the server pulls, restarts the service. No complex CI/CD pipelines, no build artifacts to manage, no deployment strategies to worry about. It just works.

Debugging

When something breaks (and it will), debugging is trivial. There's no transpiled code to trace through, no framework magic to understand, no build tool configuration to troubleshoot. The error messages point directly to the problem.

Maintenance and Technical Debt

This is the part that most developers don't talk about enough. Every dependency you add is technical debt. Every framework version is a ticking time bomb of breaking changes.

With this minimal approach:

  • Fewer dependencies: Less to update, less to break
  • Standard technologies: HTML, CSS, Python, and PostgreSQL aren't going anywhere
  • No framework churn: I'm not rewriting everything every 2 years because the ecosystem moved on
  • Easier onboarding: Any developer who knows basic web technologies can contribute

I spend my time writing content and adding features, not fighting with tooling and dependencies.

What I Actually Sacrificed

Let's be honest - there are trade-offs. Here's what I gave up by avoiding the modern framework ecosystem:

  • Complex client-side interactions: No infinite scroll, no live updates, no complex state management
  • Component reusability: I write more basic HTML instead of composing React components
  • Hot module replacement: I refresh the browser manually (takes 0.2 seconds, somehow I survive)
  • TypeScript: I don't have compile-time type checking on the frontend

But here's the thing: I don't actually need any of that for a blog. Users don't care if I'm using the latest framework. They care if the site loads fast, looks good, and the content is easy to read.

The Psychological Benefits

There's something incredibly liberating about working with simple tools. I don't have decision fatigue about which state management library to use. I don't worry about bundle size optimization strategies. I don't spend hours configuring build tools.

I just build things.

This mental bandwidth goes into writing better content, designing better features, and actually shipping things instead of endlessly configuring tooling.

When You DO Need the Frameworks

Let me be clear: frameworks aren't evil. They solve real problems for real applications. If you're building:

  • A complex single-page application with lots of client-side state
  • Real-time collaborative tools
  • Data-heavy dashboards with complex visualizations
  • Anything that genuinely needs sophisticated client-side logic

Then yes, use React or Vue or whatever. The frameworks exist for good reasons.

But if you're building a blog, a documentation site, a portfolio, or any primarily content-focused site? You probably don't need them. And you'll be better off without them.

Lessons Learned

After months of running this simple stack, here are my key takeaways:

  • Start simple, add complexity only when needed: You can always add frameworks later if requirements change
  • Performance is a feature: Users notice fast sites, even if they don't consciously realize why
  • Maintenance costs are real: Every dependency is a commitment to future maintenance work
  • Simplicity scales: As this blog grows, the simple architecture continues to work well
  • Developer happiness matters: Working with simple tools is more enjoyable

The Bigger Picture

This isn't just about my blog. It's about a broader shift happening in web development. More developers are realizing that the complexity treadmill isn't sustainable. Projects like HTMX, Alpine.js, and the return of server-side rendering show that the industry is reconsidering the "JavaScript all the things" approach.

We built websites before React existed. We can still build great websites without needing a framework for everything.

Conclusion

Cutting out the bloat wasn't just a philosophical stance - it delivered real, measurable benefits. The site is faster, easier to maintain, simpler to deploy, and more enjoyable to work on.

Could I have achieved similar results with a modern framework? Maybe. But it would have required more effort, more expertise, more tooling, and more ongoing maintenance. The simple approach just works, and it keeps working without constant attention.

As I mentioned in this blog framework ain't perfect, there are still rough edges and things I want to improve. But the foundation is solid, and that's because it's built on boring, proven technology that just works.

So if you're starting a new project, ask yourself: do you really need that framework? Or are you reaching for complexity out of habit? Sometimes the best solution is the simplest one.

And if you don't believe me, just look at the Lighthouse scores. The numbers speak for themselves.
